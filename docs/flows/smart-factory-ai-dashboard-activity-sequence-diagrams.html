Activity and Sequence Diagrams

These diagrams describe the main operational behavior of the Smart Factory AI Dashboard. They are designed to support implementation planning for the React frontend and C# ASP.NET Core backend.

Overall Activity Diagram

This activity diagram shows the main user journey from login to dashboard monitoring, AI analysis, operational action, and reporting update.

Global flow
1. User opens app User accesses the Smart Factory AI Dashboard through the browser.
2. Login and role check System verifies account and loads role-based permissions.
3. Load dashboard KPIs, production status, warehouse signals, workforce data, and alerts are displayed.
4. Detect issue? System checks whether any module has warning or critical status.
5. AI recommendation System creates suggestions for safety, productivity, warehouse, or staffing decisions.
6. User action and record User resolves, approves, rejects, escalates, or exports. The result is saved for reporting.
Normal path

If no critical issue exists, the user continues monitoring and reviewing dashboard metrics.

Warning path

If a warning appears, the user opens the related module and decides whether action is needed.

Critical path

If a critical alert appears, the system highlights it on the dashboard and routes it to the responsible role.

Detailed Demo Scenarios

These scenarios can be used for presentation, testing, and feature planning. Each scenario includes preconditions, main path, alternate path, and expected result.

Use case scripts
Scenario 1

Critical Safety Alert During Operation

Precondition

Line B is running normally. Robot Cell 2 is marked as restricted. Safety officer is logged in.

Main path

AI camera detects a worker entering Robot Cell 2, system classifies the event as critical, creates an alert, and shows it on Dashboard.

User action

Safety officer opens alert detail, reviews timestamp and location, dispatches safety staff, adds note, and marks incident resolved.

Alternate path

If camera confidence is low, the alert is marked needs review instead of critical until the officer confirms it.

Expected result

Alert status becomes resolved, action history is saved, and safety report counts one restricted-zone incident.

Scenario 2

Line Efficiency Drops Below Target

Precondition

Line C has target output of 2,000 units but actual output is below plan and efficiency is 78%.

Main path

System marks Line C as warning and AI checks downtime, defect count, staffing, and material status.

User action

Production manager opens Line C detail and applies AI suggestion to request two additional operators from Line 5.

Alternate path

If the cause is machine downtime instead of staffing, system suggests creating a maintenance request.

Expected result

Workforce module receives staffing request, Line C recommendation status changes to applied, and dashboard updates risk level.

Scenario 3

Warehouse Item Stored in Wrong Zone

Precondition

Batch RM-204 should be stored in Raw Material Zone A but is detected in Finished Goods Zone C.

Main path

System creates wrong placement alert, shows expected and actual zone, and notifies warehouse manager.

User action

Warehouse manager assigns staff to verify, confirms the item is misplaced, and records movement back to Zone A.

Alternate path

If the item was moved intentionally but not recorded, manager updates movement history instead of moving the item.

Expected result

Alert is resolved, goods movement history is updated, and warehouse occupancy statistics are recalculated.

Scenario 4

AI Suggests Night Shift Overtime

Precondition

Daily target is behind by 600 units. Night shift has 2 missing operators and 8 available employees.

Main path

Manager opens Workforce page, enters target, and AI suggests moving 2 operators plus 1.5 hours overtime.

User action

Manager reviews cost impact, modifies one employee assignment, approves the plan, and publishes schedule.

Alternate path

If overtime limit is exceeded, system suggests reducing target or moving workers from lower-priority lines.

Expected result

Shift plan becomes published, affected employees receive notification, and overtime record appears in reports.

Scenario 5

Employee Submits Leave Request

Precondition

Employee belongs to Line A morning shift and wants to request leave for tomorrow.

Main path

Employee opens Leave Request, system prefills employee and shift data, and employee submits the form.

System check

System checks whether Line A still has enough workers if the request is approved.

Manager action

Manager approves if coverage is enough or rejects with reason if the line becomes understaffed.

Expected result

Employee receives result notification and workforce availability updates if approved.

Scenario 6

Machine Fault Report Affects Production

Precondition

Operator notices abnormal vibration on Press Unit P-14 while Line D is running.

Main path

Operator submits Machine Issue Report with machine id, line, severity, and description.

System action

System notifies production manager, marks Line D as warning, and creates maintenance task.

Alternate path

If severity is critical, Line D status changes to stopped and dashboard immediately shows production risk.

Expected result

Issue status is tracked until resolved, downtime is calculated, and production report includes the incident.
